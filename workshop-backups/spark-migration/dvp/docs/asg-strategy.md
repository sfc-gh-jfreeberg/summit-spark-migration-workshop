# ASG-Centric Strategy

How DVP skills use the Abstract Semantic Graph (ASG) as the central intermediate representation, combining deterministic algorithms with AI to maximize accuracy and efficiency.

## Core Idea

The ASG is a structured JSON representation of a workload's data flow: inputs, outputs, transformations, execution calls, and source file metadata. It supports both PySpark (Python) and Spark (Scala) workloads. It is generated by `dvp-asg-generation` from the client's original source code (`<sma-input>`), producing `XX_asg.json` (where `XX` is a short project prefix, e.g. `RBI_asg.json`).

DVP skills consume the ASG instead of raw source code. This gives deterministic algorithms a clean, structured input and gives the AI agent a dramatically smaller, more focused context.

## Three-Layer Resolution

Each DVP skill that needs to understand the workload operates in three layers:

| Layer | Coverage | Input | Role |
|-------|----------|-------|------|
| **Deterministic** | ~80% | ASG JSON | WARP tools apply rules to structured data — fast, consistent, reproducible |
| **AI + ASG** | ~15% | ASG JSON + anomaly reports | AI resolves ambiguities by reasoning over the structured graph |
| **AI + source code** | ~5% | Specific code locations (from ASG references) | AI reads targeted code sections for edge cases the graph couldn't capture |

### Why this ordering matters

1. **Deterministic first**: handles the common patterns reliably. No LLM token cost, no variability.
2. **AI on structured data**: the AI reads a JSON document, not 10,000 lines of Python. Faster, cheaper, more accurate.
3. **AI on source code as last resort**: only when the ASG lacks information. ASG nodes include `location.pathfile` and `location.lineno`, so the AI reads exactly the relevant 10-20 lines — not entire files.

### Improving over time

The 80/15/5 split is a starting point. As deterministic algorithms handle more edge cases, the distribution shifts toward 90/8/2 or better. The architecture supports this gracefully — the AI layer simply has less work to do.

## The Feedback Loop

Each WARP tool generates an **anomaly report** alongside its primary output. This report lists scenarios the deterministic algorithm couldn't handle or handled with low confidence.

```
┌──────────────────────────────────────────────────────────┐
│                    Skill Execution                       │
│                                                          │
│  1. WARP tool runs on ASG                                │
│     ├── Primary output (entrypoints.json, data_io_schema.json)  │
│     └── Anomaly report (unresolved cases, low confidence)│
│                                                          │
│  2. AI reviews anomaly report                            │
│     ├── Resolves anomalies using ASG context              │
│     ├── Falls back to source code for remaining cases    │
│     └── Updates primary output with fixes                │
│                                                          │
│  3. Anomaly patterns collected for tool improvement      │
│     └── New patterns → deterministic rules in next       │
│         version of the WARP tool                         │
└──────────────────────────────────────────────────────────┘
```

### Why real-time feedback matters

The skill execution is the **best opportunity** to resolve and learn from anomalies because:

- **Source code is available**: the client's workload code is right there in the workspace. The AI can inspect any file, any line.
- **ASG is available**: the structured graph provides context that makes the AI's analysis faster and more targeted.
- **Anomaly report is available**: the AI knows exactly which cases to focus on — no guessing.
- **Afterward is too late**: once the skill execution ends, access to the client's code may be lost. Making improvements without the code context is much harder (or impossible).

### Dual benefit

Each anomaly resolution serves two purposes:

1. **Immediate**: the client's DVP output is complete and accurate for this run.
2. **Long-term**: the anomaly pattern is captured as input for improving the deterministic tool, so future runs handle it automatically.

## Embedding the Tools

The WARP tools that consume the ASG are embedded directly in the skills repo. `warp_core` is shared across all skills inside the orchestrator, while each skill has its own specific package:

```
skills/spark-migration/dvp/
├── dvp-orchestrator/
│   └── warp_core/                    (SHARED: IR models, slicer, schema, diagnostics)
├── dvp-asg-generation/warp/
│   ├── asg_pyspark/                  (PySpark AST → ASG)
│   └── asg_scala/                    (Scala Spark → ASG)
├── dvp-entrypoint-identifier/warp/
│   └── entrypoints/                  (ASG → entrypoints)
├── dvp-io-schema-identifier/warp/
│   └── data_io/                      (ASG → data_io with schemas)
└── dvp-synthetic-data-generator/warp/
    ├── synthetic_data/               (data_io → CSV test data)
    └── diagnostic_reporter/          (all WARP outputs → reports)
```

### Why embed instead of distribute separately

| Concern | Embedding solves it |
|---------|-------------------|
| Client doesn't have WARP installed | Source is in the skill — nothing to install beyond `pip install pydantic astroid sqlglot` |
| Source code protection | Not a goal — these are internal tools; the algorithm value is in the combination, not individual files |
| Version coupling | Skill and tools evolve together — no mismatched versions |
| AI can improve the tools | Since the tool source is in the workspace, the AI can read, understand, and even propose fixes to the deterministic algorithms during execution |

### External dependencies

All dependencies are public PyPI packages:

| Package | Used by | Purpose |
|---------|---------|---------|
| `pydantic` | `warp_core` (all skills) | Data models, validation |
| `astroid` | `asg_pyspark` only | Python variable inference (resolves `table_name = "raw_transactions"` → string value) |
| `sqlglot` | `asg_pyspark`, `asg_scala` | SQL parsing for embedded SQL statements |

## ASG Generation

The ASG can be generated in two ways:

### Server-side (preferred)

The SMA pipeline already processes the client's source code. Adding `warp asg` to the pipeline produces `XX_asg.json` as an additional SMA output artifact. The DVP skills on the client side consume it directly — zero WARP dependencies needed.

### Client-side (fallback)

If no ASG arrives with the SMA output, the `dvp-asg-generation` skill generates it locally using the embedded `asg_pyspark` or `asg_scala` code. This requires `pip install pydantic astroid sqlglot`.

### Graceful degradation

If neither option is available (no ASG from SMA, no dependencies installed), skills fall back to pure AI-based detection — scanning source code directly. This is slower and less consistent but still functional.

## WARP Code Synchronization

The WARP tools are developed in their own git repo (`warp-suite`) and embedded as vendored copies in the skills repo (`migrations-ai-skills`). Improvements flow in both directions:

- **Upstream (warp-suite → skills)**: WARP devs ship new features, bug fixes, and algorithmic improvements
- **Downstream (skills → warp-suite)**: AI discovers fixes during real client runs via anomaly resolution

This creates a synchronization challenge. Below are the evaluated strategies and the recommended approach.

### Strategy Options

#### Option A: Git Subtree

Embed WARP packages as a git subtree inside the skills repo.

- `git subtree pull` absorbs upstream dev improvements
- `git subtree push` sends skill-side fixes back as a branch on warp-suite (becomes a PR)
- Bidirectional by design, full history preserved
- Downside: merge conflicts when both sides touch the same file, `subtree push` creates messy commit history, and the git commands are unfamiliar to most developers

#### Option B: Git Submodule + PR Workflow

Reference warp-suite as a git submodule pinned to a specific commit.

- `git submodule update --remote` pulls latest
- AI fixes are captured as patch files (not direct edits), then submitted as PRs against warp-suite
- Clean separation — skills repo never has "dirty" warp code
- Downside: **breaks the real-time feedback loop** — the AI cannot modify the embedded WARP code during execution because the submodule is read-only. Fixes are deferred, not instant

#### Option C: Vendored Copy + Manual Sync Script

Keep a plain copy of the WARP packages. A shell script manages diffs and copies in both directions.

- Simplest mental model — the code is just regular files
- AI can freely modify during execution
- Downside: manual process, risk of drift if syncs are infrequent, no automatic conflict detection

#### Option D: Vendored Copy + Automated Patch Pipeline (recommended)

Combines vendored copy for simplicity with automated classification and PR creation for the feedback loop.

```
skills/spark-migration/dvp/dvp-orchestrator/warp_core/              ← shared vendored copy (committed to git)
skills/spark-migration/dvp/dvp-asg-generation/warp/asg_pyspark/     ← skill-specific vendored copy
skills/spark-migration/dvp/dvp-asg-generation/warp/asg_scala/       ← skill-specific vendored copy
```

### Recommended Strategy: Automated Patch Pipeline

#### Upstream → Skills (periodic)

1. CI job (or manual trigger) compares `.warp-base-ref` against warp-suite `main`
2. If new commits exist, creates a PR on the skills repo with the updated vendored copy
3. Reviewer checks for conflicts with any local modifications, merges
4. `.warp-base-ref` is updated to the new commit SHA

#### Skills → Upstream (per-run, automated)

1. During skill execution, the AI modifies WARP code to resolve anomalies
2. After the skill run completes, a `git diff` of the `warp/` folders captures all changes
3. Each change is classified by the AI:
   - **General fix** (improves the algorithm for all workloads) → saved as a patch with context
   - **Client-specific workaround** (only relevant for this workload) → logged but not pushed upstream
4. General patches are bundled into a PR on warp-suite, including the anomaly context that triggered the fix

```
┌───────────────────────────────────────────────────────────────┐
│                     Skill Execution                           │
│                                                               │
│  1. Load vendored WARP code from warp/                         │
│  2. Run deterministic tool → primary output + anomaly report  │
│  3. AI resolves anomalies (may modify WARP code in warp/)     │
│  4. Primary output completed for the client                   │
│  5. Diff warp/ against .warp-base-ref                          │
│  6. For each modification:                                    │
│     ├── AI classifies: general vs. client-specific            │
│     ├── General → save patch + anomaly description            │
│     └── Client-specific → log only                            │
│  7. Post-run: general patches → PR on warp-suite              │
└───────────────────────────────────────────────────────────────┘
```

#### Why this approach wins

| Criteria | Subtree | Submodule | Manual Sync | Automated Patches |
|----------|---------|-----------|-------------|-------------------|
| AI can modify WARP code at runtime | Yes | No | Yes | **Yes** |
| Upstream pull | `subtree pull` | `submodule update` | Script | **Automated PR** |
| Downstream push (feedback) | `subtree push` | External PRs | Script | **Auto-classified patches → PRs** |
| Filters client-specific fixes | No | No | No | **Yes** |
| Developer familiarity | Low | Medium | High | **High** (just files + PRs) |
| Real-time feedback loop | Yes | No | Yes | **Yes** |

#### The flywheel effect

Over time, this creates a self-improving cycle:

```
More client runs
      │
      ▼
More anomaly patches with real-world context
      │
      ▼
Better deterministic WARP tools
      │
      ▼
Fewer anomalies per run
      │
      ▼
Less AI work → faster, cheaper, more consistent runs
      │
      ▼
More clients can be served → more client runs → ...
```

Each patch that flows upstream carries the anomaly context: what workload triggered it, what the ASG looked like, what the fix does, and why the AI classified it as generalizable. This context is invaluable for WARP developers — it's not just a code diff, it's a documented edge case with a working solution.
