# DVP Test Suite — How to Run

## Getting Started

Follow these steps in order to run the validation process:

### Step 1: Install dependencies

Requires Python 3.10+.

**Option A: uv (recommended)**

```bash
cd dvp/03-tests
uv venv --python 3.10
source .venv/bin/activate   # Windows: .venv\Scripts\activate
uv pip install -r requirements.txt
```

> If you don't have `uv` installed: `curl -LsSf https://astral.sh/uv/install.sh | sh`

**Option B: venv**

```bash
cd dvp/03-tests
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

**Option C: conda**

```bash
cd dvp/03-tests
conda create -n dvp-tests python=3.10 -y   # skip if env already exists
conda activate dvp-tests
pip install -r requirements.txt
```

### Step 2: Verify Java (for source tests)

Source tests use PySpark, which requires Java 17+ (PySpark starts a local JVM).

```bash
java -version
# If not installed or too old:
brew install --cask temurin@17
export JAVA_HOME=$(/usr/libexec/java_home -v 17)
```

Add the `JAVA_HOME` export to `~/.zshrc` for persistence.

### Step 3: Run source tests to generate baselines

Source tests execute the original PySpark code and save baseline CSV outputs to `data/expected_output/`.

```bash
pytest source/ -v
```

After running, verify that CSV files were created:

```bash
ls data/expected_output/
```

### Step 4: Configure Snowflake connection (for migrated tests)

The test config (`dvp/03-tests/config.py`) automatically reads your default connection from `~/.snowflake/connections.toml`. If you already have a working Snowflake TOML config, no extra setup is needed.

To override or if you don't have a TOML config, set environment variables:

```bash
export SNOWFLAKE_CONNECTION_NAME=my_connection
export SNOWFLAKE_TEST_DATABASE=MY_DB
export SNOWFLAKE_TEST_SCHEMA=MY_SCHEMA
export SNOWFLAKE_TEST_ROLE=MY_ROLE
export SNOWFLAKE_TEST_STAGE=MY_STAGE
```

### Step 5: Run migrated tests

Migrated tests run the converted Snowpark code on Snowflake and compare results against the baselines from Step 3.

```bash
pytest migrated/ -v
```

Or for SCOS migrations:

```bash
pytest migrated_scos/ -v
```

### Step 6: Review results in the dashboard

Test results are automatically recorded in `sma_storage.sqlite3` when tests run. Open the SMA Dashboard to review pass/fail status per entrypoint:

```bash
python sma-dashboard/start_server.py
```

Navigate to the **Test Tracker** module to see:
- Overall pass/fail status per entrypoint
- Detailed test run history (click "View Runs" on any test)
- Error messages for failed tests

To manually register tests after adding new test files:

```bash
python3 <skills_path>/dvp/dvp-test-setup-generator/scripts/register_tests.py \
  --workload-path <workload_path>
```

### Step 7: Fix failures and re-run

For each failing test:

1. Read the error message in the dashboard or pytest output
2. Fix the migrated code in `dvp/02-migrated/` (or `dvp/02-migrated_scos/`)
3. Re-run only the failing test to verify:
   ```bash
   pytest migrated/test_MyWorkload.py -v
   ```
4. Once all tests pass, the migration for that entrypoint is validated

## What Gets Tested

For each entrypoint detected in `entrypoints.json` (generated by the DVP Entrypoint Identifier skill), four tests are generated, organized in two categories:

### Execution validation

| Test | What it validates |
|------|-------------------|
| `test_runs_without_error` | The workload executes without raising exceptions. This is the most basic check — if the code crashes, this test fails. |

This test applies to every entrypoint, regardless of whether it produces outputs.

### Output data validation

| Test | What it validates |
|------|-------------------|
| `test_all_outputs_have_data` | Every declared output (file or table) exists and contains at least one row. |
| `test_all_outputs_match_baseline_row_count` | The number of rows in each output matches the expected baseline CSV. Catches missing or duplicated rows. |
| `test_all_outputs_match_baseline` | Full data comparison — schema and values must match the baseline CSV. Catches column renames, type changes, and incorrect values. |

These three tests only apply to entrypoints that declare outputs (`OUTPUT_FILES` or `OUTPUT_TABLES`). If an entrypoint has no declared outputs (e.g., it only calls `.show()`), they iterate over an empty list and pass automatically.

### Source vs. Migrated tests

- **Source tests** (`source/test_*.py`) run the original PySpark code locally. Their purpose is to generate baseline CSV files in `data/expected_output/` that serve as the reference for comparison.
- **Migrated tests** (`migrated/test_*.py` or `migrated_scos/test_*.py`) run the converted Snowpark code on Snowflake. They compare Snowflake results against the baselines generated by source tests.

Run source tests first to generate baselines, then migrated tests to validate the conversion.

```
                          dvp/03-tests/
                               │
              ┌────────────────┴────────────────┐
              │                                 │
         source/                          migrated/
     (PySpark local)                   (Snowpark on Snowflake)
              │                                 │
     ┌────────┴────────┐              ┌────────┴────────┐
     │  Execution      │              │  Execution      │
     │  Validation     │              │  Validation     │
     │                 │              │                 │
     │  test_runs_     │              │  test_runs_     │
     │  without_error  │              │  without_error  │
     └────────┬────────┘              └────────┬────────┘
              │                                │
     ┌────────┴────────┐              ┌────────┴────────┐
     │  Output Data    │              │  Output Data    │
     │  Validation     │              │  Validation     │
     │                 │              │                 │
     │  _have_data     │              │  _have_data     │
     │  _row_count     │              │  _row_count     │
     │  _match_baseline│              │  _match_baseline│
     └────────┬────────┘              └────────┬────────┘
              │                                │
              ▼                                ▼
   data/expected_output/              Compare against baselines
     (baseline CSVs)                  from source tests
              │                                │
              └───────────── ◄─────────────────┘
                    Baselines are the
                    reference for comparison
```

## Quick Reference

Common pytest commands (run from `dvp/03-tests/`):

```bash
pytest source/ -v                        # Source tests only
pytest migrated/ -v                      # Migrated tests only
pytest source/test_MyWorkload.py -v      # Single test file
pytest -v                                # All tests
```

## Troubleshooting

### `UnsupportedClassVersionError: class file version 61.0`

Your Java version is too old. PySpark requires Java 17+.

```bash
export JAVA_HOME=$(/usr/libexec/java_home -v 17)
```

### `JAVA_GATEWAY_EXITED`

Java is not installed or `JAVA_HOME` is not set. Install Java 17+ and set the environment variable.

### `ModuleNotFoundError: No module named 'pyspark'`

Install dependencies:

```bash
pip install -r requirements.txt
```

### `snowflake.connector.errors.DatabaseError: connection refused`

Check your Snowflake connection in `~/.snowflake/connections.toml` or environment variables. Verify the connection works:

```bash
python -c "import snowflake.connector; c = snowflake.connector.connect(connection_name='YOUR_CONNECTION'); print('OK'); c.close()"
```

### `FileNotFoundError: data_io_schema.json`

The synthetic data generation step has not run yet. Go back to the DVP orchestrator and run the synthetic data generator skill.

## VS Code Integration

Tests are automatically configured for VS Code Test Explorer via `.vscode/settings.json` at the workload root. After opening the workload folder in VS Code:

1. Select the correct Python interpreter (Cmd+Shift+P → "Python: Select Interpreter")
2. Open the Test Explorer (flask icon in the sidebar)
3. Tests should appear organized by `source/` and `migrated/`

**If the interpreter doesn't appear in the list:**

1. Select "Enter interpreter path..." at the bottom of the list
2. Navigate to the Python binary inside your environment:
   - **uv / venv**: `dvp/03-tests/.venv/bin/python` (macOS/Linux) or `dvp/03-tests\.venv\Scripts\python.exe` (Windows)
   - **conda**: `~/anaconda3/envs/dvp-tests/bin/python` (macOS/Linux) or `C:\Users\<you>\anaconda3\envs\dvp-tests\python.exe` (Windows)

> **Tip:** If VS Code still doesn't detect `.venv`, make sure you opened the **workload root folder** (the one containing `dvp/`), not a parent directory. VS Code only auto-discovers `.venv` directories within the opened workspace.

If tests don't appear in the Test Explorer, verify that `.vscode/settings.json` contains:

```json
{
    "python.testing.pytestEnabled": true,
    "python.testing.pytestArgs": [
        "dvp/03-tests"
    ]
}
```
